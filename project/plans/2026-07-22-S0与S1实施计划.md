# S0 项目基线与 S1 最小运行骨架实施计划

> **供智能体执行者使用：** 实施时优先使用 `superpowers:subagent-driven-development`，或使用 `superpowers:executing-plans` 逐任务执行；若相关技能不可用，则在当前会话内按相同的测试驱动步骤执行。所有步骤使用复选框跟踪。

**目标：** 建立结构化项目管理与中文产品文档，并交付一个完全依赖合成数据、可通过命令行和 HTTP 调用、能把病例订单保存为固定结构病例包的最小纵向切片。

**架构：** 使用顶层 Python 包管理，后端代码位于 `apps/api/src/medicine_agents`，通过领域模型、端口接口、文件仓库和应用工作流分层。S1 不调用商业模型、不导入真实医学资料；所有输出写入被 Git 忽略的 `data/runtime/`。前端只建立边界说明，正式界面在 S6 独立计划中实现。

**技术栈：** Python 3.11+、FastAPI、Pydantic v2、Uvicorn、pytest、HTTPX、Ruff、JSON 文件存储、PowerShell 7。

---

## 计划覆盖范围

本计划只交付设计规格中的 S0 和 S1：

- S0：仓库、项目管理、中文 PRD、质量评价方案、发布治理和医学输入台账；
- S1：合成订单 → 固定病例包 → 原子文件保存 → 命令行/HTTP 返回结果；
- 不实现 RAG、商业模型调用、真实医学资料导入、自动质控、发布状态转换或 React 页面。

S2–S9 分别建立独立实施计划，以每阶段都可体验、可验收为原则推进。

- [ ] **步骤 3：写入指标定义和样本设计**

方案采用以下固定定义：

| 指标 | 分子/分母或判定 | MVP 门槛 | 正式评价门槛 |
|---|---|---:|---:|
| 原子事实准确率 | 正确可核验事实数/全部可核验事实数 | ≥90% | ≥90% |
| 无依据陈述率 | 无来源、来源冲突或虚构事实数/全部可核验事实数 | ≤5% | ≤5%，目标≤3% |
| 逻辑自洽率 | 无重大时间线、因果链或诊断依据冲突病例数/评价病例数 | ≥90% | ≥95% |
| 三轮通过率 | 三轮内自动质控通过病例数/生成病例数 | 记录基线 | ≥80% |
| 人工介入率 | 三轮未通过病例数/生成病例数 | 记录基线 | ≤20% |
| 严重医学错误 | 可能形成错误诊疗认知的错误 | 发布病例为 0 | 发布病例为 0 |

MVP 的 30 个病例全部进行至少一名医学人员审核；正式 100 例评价集按病种和难度分层抽样，由三名独立专家评分。争议事实由预先指定的第三方规则裁决，不由生产质控智能体自评。

- [ ] **步骤 4：写入统计方法**

明确使用 `ICC(2,k)` 评价连续/量表平均评分一致性，ICC 及 95% CI 下限达到 0.75 为可接受；分类错误标签使用 Fleiss' Kappa，目标不低于 0.80。AI 与人工病例的质量和难度比较使用预设等效界值及 TOST/置信区间，不以 `P>0.05` 代替等效性证明。

教学研究中，OSCE 总分为主要结局；满意度和分维度得分为次要结局；Cohen's d=0.6 只用于样本量估算。统计方案必须涵盖基线比较、正态性、方差齐性、意向性分析、符合方案分析、缺失数据、多重比较和效应量置信区间。

- [ ] **步骤 5：写入版本冻结与数据集隔离**

开发集、回归集、Judge 校准集和最终盲测集互相隔离。进入正式教学验证前冻结代码版本、模型标识、Prompt、知识库清单和质控规则；研究期间只允许修复严重安全问题，且必须记录偏离。
