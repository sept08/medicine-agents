# 医学教学病例智能体产品需求文档

## 1. 文档信息

- 文档状态：S1 产品基线
- 产品形态：面向单操作者的本地优先 Web 应用
- 当前目标：以最小成本跑通病例教学问题答案包的生成、审核、发布与导出主流程
- 使用边界：仅供医学教学与项目验证，不用于临床诊断、治疗决策或患者服务

## 2. 背景与目标

医学专业人员需要把教材、指南和脱敏病例转化为结构化教学病例，并为病例生成分层问题、参考答案和依据。现有人工流程重复度高，过程证据与版本不易追踪。

本产品以“可体验、可验证、可替换”为原则，先支持一名操作者在单机完成业务闭环，再根据验证结果决定是否迁移到云端数据库、对象存储、任务队列和企业级安全服务。

核心目标：生成结构化病例与教学问题答案包；保留医学陈述的可核查依据；通过自动质控、人工编辑和医学审核后发布；以文件优先方式保留版本和运行记录；隔离可替换的模型、检索、存储和任务组件。

## 3. 用户与职责

### 3.1 单操作者

创建病例订单、导入资料、触发生成、查看质控、编辑内容、发起医学审核、发布和导出成果。MVP 不建设复杂账号、组织和权限体系。

### 3.2 医学审核者

提供经确认可用于项目的教材、指南和脱敏病例；检查医学准确性、证据一致性和教学适用性；批准、驳回或要求修改待发布内容。

### 3.3 项目负责人

协调医学输入、确认阶段验收结果、处理范围和质量门槛变更，并决定是否进入下一阶段。

## 4. 冻结规模与口径

项目统一采用以下口径，任何变化必须记录到决策日志和变更日志：

- 首次可运行验证：1 个病种，3 个合成或演示病例问题答案包。
- MVP 验收：6 个病种，共 30 个通过医学审核的问题答案包。
- 项目最终验证：20 个病种，共 100 个通过医学审核的问题答案包。
- 可供筛选、生成与研究使用的脱敏病例语料目标：覆盖上述 20 个病种，累计 500 例；语料数量不等同于最终发布包数量。

MVP 冻结病种为糖尿病肾病、慢性肾脏病、急性肾损伤、慢性心力衰竭、急性心肌梗死和高血压急症。具体编码以 `config/diseases/mvp.yaml` 为准。

## 5. MVP 范围

### 5.1 包含范围

- 标准化病例订单的创建、校验和保存。
- 教材、指南与脱敏病例的本地导入登记。
- 基于订单和证据的病例问题答案包生成。
- 基础与进阶两类难度，以及教学目标配置。
- 自动质控，最多三轮修订并保留每轮结果。
- 人工编辑、医学审核、批准、发布、撤回和退役。
- 已发布病例库浏览、筛选、查看和文件导出。
- 生成过程、模型调用、证据引用、审核意见和版本的可追踪记录。
- 仅在本地私有目录保存医学原始资料和运行产物。

### 5.2 暂不包含

- 多租户、复杂角色权限、开放注册和公网大规模服务。
- 面向患者的临床建议或自动诊疗。
- RDS、OSS、WAF、分布式队列和高可用集群的强制部署。
- 医院信息系统或电子病历系统的实时集成。
- 自动替代医学专业人员完成最终发布审批。

## 6. 业务主流程

1. 操作者登记资料来源并把授权材料放入不受 Git 管理的私有目录。
2. 操作者创建病例订单，选择病种、难度、受众和教学目标。
3. 系统检索相关病例与知识证据，并形成可引用证据集合。
4. 生成组件输出结构化病例、分层问题、参考答案和证据引用。
5. 质控组件检查结构完整性、证据覆盖、医学风险和内容一致性。
6. 未通过时自动修订，最多三轮；仍不通过则转人工处理。
7. 操作者编辑后提交医学审核。
8. 医学审核者批准、驳回或要求修改。
9. 仅已批准版本可发布；发布后进入病例库并可导出。
10. 发现问题时可撤回或退役，修改后必须重新审核。

## 7. 功能需求

### 7.1 病例订单

必填项为病种编码、难度、目标受众和至少一个教学目标。系统分配唯一标识并记录创建时间；无效病种、空教学目标或不支持的难度必须被拒绝。

### 7.2 资料与证据

每项资料记录来源类型、标题、版本或发布日期、授权状态和本地路径。真实医学资料不得进入版本库；仓库仅提供合成样例。关键医学事实应关联一个或多个证据标识；无法确认的内容必须明确标记，不得伪造引用。

### 7.3 病例问题答案包

病例内容至少包含主诉、现病史、检查摘要、诊断摘要和教学说明。问题至少包含层级、题干、参考答案和证据标识。每个包记录订单标识、病例标识、是否为合成数据、状态、创建时间和版本。

### 7.4 自动质控与修订

检查必填字段、证据引用、问题答案一致性、潜在严重医学错误和敏感信息风险。每轮输出通过状态、规则结果、问题列表和修订建议。自动修订不得超过三轮，且不得覆盖既有版本和日志。

### 7.5 编辑与医学审核

操作者可修改未发布版本并填写修改说明。医学审核意见必须记录审核结论、问题、修改要求和时间。内容发生实质修改后，原批准失效并重新进入审核。

### 7.6 发布治理

只有 `approved` 状态能够转为 `published`。发布版本必须展示“仅供医学教学”声明、版本号和证据概览。已发布版本不可静默覆盖；问题版本可撤回或标记为 `retired`。具体规则以《MVP 医学内容发布治理规范》为准。

### 7.7 病例库与导出

支持按病种、难度、状态和教学目标筛选；支持查看当前版本、证据概览和审核记录；支持导出机器可读 JSON，教学使用格式在后续阶段按反馈补充。

## 8. 状态模型

病例包采用：`draft → qc_passed → awaiting_medical_review → approved → published → retired`。

质控失败保留在 `draft`；医学审核驳回返回 `draft`；已发布内容修改时创建新版本，新版本从 `draft` 开始，原发布版本在新版本获批前保持可追踪。

## 9. 数据与技术基线

### 9.1 文件优先

MVP 使用 JSON、YAML 和 Markdown 保存结构化配置、病例包、质控报告、审核记录与导出物。写入采用原子替换，避免部分写入；目录和文件命名保持稳定且可迁移。

### 9.2 可替换技术接口

业务逻辑不得直接依赖本地文件或单一模型厂商，通过以下边界隔离：

- `CaseRepository`：病例包存取；后续可由本地文件替换为 RDS。
- `KnowledgeRepository`：知识与资料元数据；后续可连接数据库或知识库。
- `Retriever`：证据检索；后续可替换为向量或混合检索。
- `ModelProvider`：模型调用；后续可切换商业模型或本地模型。
- `TaskRunner`：任务执行；后续可替换为云端队列。
- `ArtifactStore`：资料与产物；后续可由本地目录替换为 OSS。
- `TraceSink`：运行与审计记录；后续可接入集中日志服务。

上述替换不得改变领域对象和已发布文件的数据契约；迁移方案在架构决策记录中维护。

### 9.3 商业模型调用边界

S0 和 S1 不调用商业模型，仅使用合成数据验证工程骨架。首次商业模型调用预计在 S2 的单病种生成验证中发生，启用前必须完成密钥配置、费用上限、模型与版本记录、请求脱敏检查和失败降级。经脱敏并获项目授权的病例可调用商业 API；未经确认的原始资料不得发送。

## 10. 非功能要求

- 跨平台：同一代码库在 Windows 和 macOS、Python 3.11 及以上可安装、测试和运行。
- 隐私：`wiki/`、私有资料、环境变量、密钥、运行数据和敏感词表均不得提交。
- 安全：日志不得记录密钥和不必要的病例原文；发布前执行敏感信息扫描。
- 可追踪：生成、质控、编辑、审核和发布关联同一病例标识和版本。
- 可恢复：单次生成失败不得破坏既有包；失败原因可查看并可重试。
- 成本：默认单人、低并发运行；商业模型调用设置单次和阶段预算阈值。
- 可测试：领域规则、文件存储、主流程、API 和跨平台脚本具有自动化测试。

## 11. 验收标准

### 11.1 首次可体验验收

- 无真实医学资料和商业模型密钥时，可从合成订单生成并保存 1 个合成病例包。
- 可通过 API 或命令行读取病例包并查看状态、问题答案和证据。
- Windows 与 macOS 的自动化测试通过。

### 11.2 MVP 验收

- 冻结的 6 个病种均能完成主流程，共形成 30 个医学审核通过的问题答案包。
- 每个发布包具有证据、质控结果、人工审核记录、版本和教学声明。
- 严重医学错误为零；其余质量指标按《质量评价与统计分析方案》执行。
- 项目负责人可按文档在新设备完成安装、配置、运行和导出。

## 12. 医学团队输入与阻塞关系

- S0–S1：无需真实资料；医学团队可并行确认病种、模板、资料授权和审核人员。
- S2：至少需要 1 个病种的指南或教材节选、3 例脱敏病例及教学目标；缺少这些输入会阻塞真实生成验证。
- S3 以后：按输入登记表分批提供其余病种资料，无需等待全部材料到齐后才开发。
- 所有输入必须满足《医学团队资料准备指南》和仓库隐私策略。

## 13. 完整性评价

本 PRD 已覆盖 MVP 开工所需的用户、范围、主流程、数据契约、质量门槛、发布治理、非功能要求和阶段验收，可作为 S1–S7 的产品基线。正式教学研究仍以独立的伦理、质量评价与统计分析方案为准，不把研究假设作为软件验收承诺。

当前仍允许在不改变核心目标的前提下细化页面交互、导出格式和具体提示词；规模、状态模型、隐私边界、发布门槛或技术替换边界的变化必须走项目变更记录。
