# 仓库数据与隐私策略

## 1. 目的

本仓库只保存软件代码、合成测试数据和不含敏感信息的项目文档。项目申请材料、私有医学资料和可识别个人身份的信息必须保留在本地，不得进入 Git 历史或远端仓库。

## 2. 强制规则

- `wiki/` 整个目录属于本地原始材料，绝对不得提交。
- 真实病历、脱敏病历、受版权保护的教材与指南、评审数据和运行产生的病例包不得提交。
- 版本库中的示例必须完全合成，不得通过复制原始资料并简单替换字段的方式构造。
- 本地原始文档中的姓名、联系方式、证件号码、签名及其他身份属性不得出现在被跟踪文件中。
- API Key 和其他密钥只能保存在被忽略的本地配置中。
- Office 与 PDF 文件默认全部忽略，因为提交前难以对其内容进行可靠检查。

## 3. 自动防护

仓库通过 `.githooks/pre-commit` 调用 `scripts/check-sensitive-content.ps1`。提交前检查会：

1. 拒绝任何位于 `wiki/` 下的暂存文件；
2. 使用本机敏感词清单扫描所有暂存文本；
3. 拒绝常见电子邮箱、手机号码和身份证号码格式。

本机敏感词清单位于 `.local/sensitive_terms.txt`，每行一个词条。该文件本身已被忽略，禁止提交。

自动检查只是最后一道防线，不能替代人工核查。每次推送前仍应检查 `git diff --cached`，确认所有示例均为合成数据，且不包含来源材料中的个人信息。

## 4. 新设备或新克隆的初始化

每位开发者在首次提交前，都必须根据其获准接触的本地资料创建 `.local/sensitive_terms.txt`。如果敏感词清单不存在或为空，提交钩子将采用失败关闭策略，拒绝提交。

## 5. 数据目录约定

- `data/samples/`：允许提交，但只能保存明确标注的合成样例。
- `data/private/`：真实或脱敏医学输入，只保存在本地。
- `data/knowledge/`：处理后的知识库、向量和索引，只保存在本地。
- `data/runtime/`：生成病例、版本、审核记录和运行日志，只保存在本地。
- `.local/`：本机配置和敏感词清单，只保存在本地。
