# MVP 医学内容发布治理规范

## 1. 目的与边界

本规范为单人运行 MVP 提供必要的医学内容发布控制，确保自动生成内容只有在完成自动质控和医学人工审核后才能进入教学病例库。它不代替所在机构的伦理、版权、数据或临床治理制度。

所有页面和导出物必须显示“仅供医学教学，不用于临床诊疗决策”。

## 2. 角色与最低分工

- 操作者：创建订单、运行生成与质控、修订内容、提交审核和执行已获批准的发布操作。
- 医学审核者：核对事实、证据、教学目标和风险，作出批准、驳回或要求修改的结论。
- 项目负责人：确认审核者资质，处理严重错误、撤回、范围变化和争议。

MVP 允许同一人承担操作和项目管理，但医学内容不能仅凭自动质控发布。正式评价时，生产审核者不能担任唯一独立评价者。

## 3. 内容状态

`draft → qc_passed → awaiting_medical_review → approved → published → retired`

- `draft`：生成或编辑中的草稿，不可用于正式教学。
- `qc_passed`：已通过当前版本自动质控，等待操作者确认提交。
- `awaiting_medical_review`：已锁定送审版本，等待医学审核。
- `approved`：医学审核者已批准当前内容与依据，可由操作者发布。
- `published`：进入教学病例库的不可静默覆盖版本。
- `retired`：已撤回或停止使用，仍保留审计记录。

质控失败保留在 `draft`。医学审核驳回返回 `draft` 并附修改要求。任何影响医学含义、答案、问题、依据或教学目标的修改都使原批准失效，必须重新审核。

## 4. 发布前门槛

发布操作仅在以下条件全部满足时允许：

1. 当前版本状态为 `approved`，批准记录可读取。
2. 必填字段完整，问题和答案均有适用的证据引用。
3. 自动质控通过，且严重医学错误为零。
4. 医学审核者已检查事实准确性、逻辑自洽、教学适用性和证据支持。
5. 敏感信息扫描通过，未发现不应发布的身份信息或密钥。
6. 资料来源授权状态允许项目内使用，版本可追踪。
7. 发布物含“仅供医学教学”声明、病例版本和发布日期。

## 5. 审核记录

每次医学审核至少记录：病例标识、内容版本、审核者项目编号、结论、问题分类、修改要求、审核时间和所依据资料版本。不得以聊天消息或口头结论替代结构化记录。

送审后内容应只读；如需修改，退回 `draft` 创建新版本。自动质控结果和历史审核意见不得覆盖或删除。

## 6. 发布、更新与退役

- 发布创建不可变版本，病例库的“当前版本”仅是指针。
- 文字格式修正也应留下更正说明；涉及医学含义的修改必须重新审核。
- 新版本发布后，旧版本保留但默认不再展示，避免既有教学记录失去依据。
- 出现错误、证据失效、指南更新或授权变化时，可立即把版本标为 `retired`。
- 退役后不得继续导出用于新教学活动；已有使用记录继续保留。

## 7. 严重事件处置

发现潜在严重医学错误时：立即停止该版本使用并退役；通知项目负责人；检查同批次、同提示词、同规则和同来源内容；记录原因、影响、修正和预防措施；修正版本完成全量质控与医学重新审核后才可再次发布。

发现隐私或授权问题时，同样立即撤回，并按仓库数据与隐私策略处理本地副本和日志。需要外部报告时由项目负责人按机构规定决定，本系统不自动对外发送。

## 8. 最小审计与留存

必须保留订单、输入资料元数据、生成版本、模型与提示词版本、质控轮次、人工修改、审核记录、发布记录和退役原因。MVP 使用本地文件保存，写入采用原子替换；备份由项目负责人按项目策略管理。

版本库只保存结构、代码、规则和合成样例，不保存真实医学资料、真实病例运行产物或身份映射。

## 9. 例外与变更

不设置“紧急绕过医学审核”的发布入口。若项目范围、状态、门槛或角色发生变化，先更新本规范和决策日志，再修改系统行为。无法满足门槛的内容只能保留为草稿或退役，不得以演示需要作为发布理由。
