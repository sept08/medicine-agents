# 文件优先的医学教学病例 MVP 设计规格

## 1. 决策摘要

项目采用“文件优先、全栈纵向切片”的实现方案。第一目标是让一名操作者能够在普通开发环境中跑通完整业务流程，并产出经过医学审核的教学病例和问题答案包。在验证核心教学价值前，暂不引入多用户管理、高并发和重型托管基础设施。

存储、模型服务、检索、任务执行和可观测性均通过稳定接口隔离。后续升级 PostgreSQL、Milvus、任务队列、对象存储或 LangFuse 时，不改变病例生成、审核与发布的领域流程。

## 2. 冻结后的交付范围

### 2.1 数量基线

| 交付层级 | 病种范围 | 审核通过的产出 | 定义 |
|---|---:|---:|---|
| 首个可运行切片 | 1 个病种 | 3 个测试病例包 | 仅用于验证技术链路，不计入正式成果 |
| MVP | 6 个病种 | 每病种 5 个，共 30 个 | 第一版可实际使用的教学病例库 |
| 项目最终成果 | 20 个病种 | 不少于 100 个 | 正式项目成果口径 |
| 最终输入语料 | 20 个病种 | 不少于 500 份脱敏病例 | 正式知识库建设口径 |

只有完成医学审核并发布的病例才计入 30/100 个正式成果。系统生成的草稿、质控失败病例和未审核病例不计入。

### 2.2 MVP 病种

MVP 覆盖三个肾内科病种和三个心内科病种：

- 糖尿病肾病；
- 慢性肾脏病；
- 急性肾损伤；
- 慢性心力衰竭；
- 急性心肌梗死；
- 高血压急症。

### 2.3 MVP 主业务流程

1. 导入已获准使用的教材、指南和脱敏病例。
2. 填写病种、难度、目标受众和教学目标，创建病例订单。
3. 检索支持证据并生成结构化病例草稿。
4. 生成阶梯式教学问题，并为每道问题生成参考答案。
5. 执行程序化检查和模型辅助质控，最多进行三轮修正。
6. 操作者编辑病例包。
7. 提交医学专业人员审核。
8. 批准、发布、撤回或归档病例版本，并导出教学包。

### 2.4 MVP 纳入范围

- 本地资料导入、规范化和索引；
- 结构化病例订单与病例包；
- 病例、问题、答案、证据和质控报告生成；
- 程序化规则与模型辅助质控；
- 编辑、审核、批准、发布、归档和版本追溯；
- JSON 与 Markdown 导出；
- 覆盖完整流程的最小浏览器界面；
- 本地运行记录、成本记录和失败追踪；
- 完全合成的测试数据与自动化测试。

### 2.5 MVP 后再考虑的能力

- 多用户认证及复杂权限管理；
- PostgreSQL、Milvus、Redis、MinIO 及云托管组件；
- GraphRAG 与 Neo4j；
- 在线 A/B 自动选优；
- 并发任务调度与批量生成；
- 复杂统计看板；
- 精美 Word/PDF 导出；
- 正式教学研究统计的系统内自动化。

## 3. 系统架构

### 3.1 运行组件

- `apps/api`：FastAPI 服务、领域模型和主业务流程。
- `apps/web`：React + Vite 的单操作者界面。
- 本地病例仓库：每个病例一个目录，保存元数据、不可变版本、审核记录和运行追踪。
- 本地知识仓库：保存规范化文本块、清单、嵌入结果和检索元数据，全部位于 Git 之外。
- 模型适配器：通过稳定的 `ModelProvider` 接口调用兼容 OpenAI 协议的商业 API。
- 进程内任务执行器：同一时间运行一个生成任务，并记录阶段性进度事件。
- 本地可观测性：使用 JSON Lines 日志和逐次运行成本摘要。

领域层不得直接依赖具体文件路径、模型厂商或检索引擎。

### 3.2 本地病例目录

```text
data/runtime/cases/<case-id>/
├─ order.json
├─ current.json
├─ versions/
│  ├─ v001.json
│  └─ v002.json
├─ reviews/
│  └─ review-001.json
├─ exports/
└─ trace.jsonl
```

文件更新采用“写入临时文件后原子替换”的方式。已发布版本不可原地修改。

### 3.3 发布状态机

```text
draft
  → qc_passed
  → awaiting_medical_review
  → approved
  → published
  → retired
```

审核拒绝时，病例携带拒绝原因返回 `draft`。修改已批准或已发布病例时，系统创建新的草稿版本，新版本必须重新质控和审核。发布时必须记录医学审核人标识和审核时间。所有导出物必须标注“仅供医学教学，不用于真实临床诊疗”。

## 4. 后续技术替换路径

| MVP 实现 | 稳定接口 | 后续替换方案 | 启动替换的条件 |
|---|---|---|---|
| JSON 病例目录 | `CaseRepository` | PostgreSQL | 出现多用户写入、复杂查询或文件锁冲突 |
| 本地资料与索引 | `KnowledgeRepository`、`Retriever` | Milvus 或 pgvector，并配合对象存储 | 语料规模或检索延迟超过约定指标 |
| 进程内任务执行 | `TaskRunner` | Redis + Celery 或其他任务队列 | 需要并发生成、后台运行或可靠重试 |
| 本地导出目录 | `ArtifactStore` | MinIO 或云对象存储 | 需要远程访问、集中留存或大文件管理 |
| 直接模型适配器 | `ModelProvider` | LiteLLM 或获批的统一模型网关 | 需要统一配额、路由、审计或故障转移 |
| JSONL 运行追踪 | `TraceSink` | LangFuse 或 OpenTelemetry 后端 | 需要跨运行分析和共享监控 |
| 本地配置中的操作者身份 | `IdentityContext` | 登录认证与基于角色的权限控制 | 引入第二名实际操作用户 |
| 内存中的检索编排 | `Retriever` | BM25/向量混合服务及可选 GraphRAG | 质量测量证明简单检索无法达标 |

所有升级都必须由实际指标触发，不因追求架构完整而提前引入。

## 5. 数据与版本库边界

- `wiki/`、私有资料、知识库索引、运行产物、本机敏感词清单和密钥一律不得提交。
- 只有完全合成的测试样例可以进入版本库。
- 系统将“已经脱敏并获准使用”视为导入前提，并在本地资料清单中记录审批状态。
- 经确认允许后，模型调用可以使用脱敏内容；导入真实资料前仍需由操作者确认机构制度及模型服务条款。
- 每个病例版本必须记录来源标识、模型配置、Prompt 版本、知识库清单版本和质控规则版本。

## 6. 异常与失败处理

- 使用 Pydantic Schema 校验订单、病例、问题、审核记录和状态转换。
- 模型请求采用有限次数重试和指数退避，并保存最终失败原因。
- 失败任务绝不能产生 `approved` 或 `published` 状态。
- 允许保留部分运行产物用于诊断，但不得进入正式病例库。
- 三轮质控仍不通过时停止自动修正，转入人工处理。
- 文件写入必须原子化，状态转换前检查当前版本，防止覆盖新版本。
- 面向用户的错误信息只显示可执行摘要和运行编号，不显示密钥及敏感原文。

## 7. 质量评价与统计设计

项目另设《质量评价与统计分析方案》，覆盖四个层面：

1. 系统质量：完成率、Schema 有效率、生成耗时、成本和修正轮次。
2. 医学内容质量：原子事实准确率、无依据陈述率、逻辑自洽率和教学目标契合度。
3. AI 病例与人工病例等效性：预设等效界值，使用置信区间或 TOST 等效性检验。
4. 教学效果：OSCE 为主要结局，满意度和质性反馈为次要结局。

生产链路中的质控智能体不能成为自身产出的唯一评价者。MVP 发布必须经过医学人员审核。正式评价使用冻结版本、分层样本和独立评审。预期效应量只用于研究样本量估算，不作为产品必须实现的验收承诺。

## 8. 测试策略

- Schema 测试覆盖合法与非法领域对象。
- 状态机测试覆盖所有允许和禁止的转换。
- 文件仓库测试验证原子写入、版本不可变和中断恢复。
- 模型适配器契约测试使用合成响应，不依赖真实 API Key。
- 检索测试使用具有确定预期证据的小型合成语料。
- 工作流集成测试离线跑通“订单→病例→问题答案→待审核”。
- 浏览器验收测试覆盖单用户主流程及关键失败状态。
- 医学内容质量使用 Git 之外的获准资料和审核表单独立验证。

每个开发切片从一个可执行的失败测试开始，以一个用户可体验的验收结果结束。

## 9. 项目目录结构

```text
medicine-agents/
├─ apps/                 # 可执行产品代码
├─ config/               # 可提交的非敏感 Schema、Prompt 和规则
├─ data/                 # 本地数据边界，私有资料和运行内容被忽略
├─ docs/                 # 产品、架构、质量、治理与使用说明
├─ project/              # 当前状态、路线图、门禁、输入、风险与决策
├─ scripts/              # 初始化、校验、导入、备份和维护脚本
├─ tests/                # 跨组件测试与合成测试数据
└─ wiki/                 # 本地原始资料，整目录忽略
```

`project/CURRENT_STATUS.md` 是项目管理总入口，必须显示当前阶段、验收目标、已完成工作、活动阻塞项、等待医学团队提供的资料、下一步行动和最新可体验入口。

## 10. 逐步可体验的阶段

| 阶段 | 可体验结果 | 医学输入依赖 |
|---|---|---|
| S0 仓库与项目基线 | 可导航的项目、范围、治理和状态文档 | 确认 MVP 病种；不阻塞脚手架 |
| S1 最小运行骨架 | 合成订单生成固定且符合 Schema 的病例包 | 无 |
| S2 单病种生成 | 使用获准资料生成一个病种的病例 | 一个病种的教材/指南及 3–5 份脱敏病例；只阻塞医学有效性验证 |
| S3 问题答案包 | 生成阶梯问题和有证据支撑的答案 | 教学目标及 5–10 组审核过的问题答案示例 |
| S4 质控闭环 | 执行规则、最多三轮修正并输出质控报告 | 医学规则和代表性错误样例 |
| S5 发布治理 | 编辑、审核、批准、发布、归档和版本追溯 | 指定审核人及最小审核量表 |
| S6 Web MVP | 在浏览器完成完整业务流程 | 持续提供可用性反馈 |
| S7 六病种 MVP | 形成 30 个审核发布病例包 | 六病种资料及逐例审核 |
| S8 正式评价 | 冻结 100 例评价集并准备教学研究 | 专家、伦理、量表、教学排期和参与者 |
| S9 成果整理 | 项目报告、软著及论文素材 | 最终统计结果和专业审核 |

医学资料准备作为与开发并行的工作流管理。只有缺少某项输入会导致阶段无法达到其验收结果时，才把该输入标为阻塞。

## 11. 设计变更规则

本设计是实施计划的基线。冻结范围、数据边界、发布规则或技术替换接口发生变化时，必须在项目决策日志和变更日志中记录原因、影响和批准结论。
