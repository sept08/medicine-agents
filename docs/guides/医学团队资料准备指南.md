# 医学团队资料准备指南

## 1. 目的

本指南说明医学专业同事需要提供什么、如何脱敏、何时会阻塞开发，以及哪些工作可以与功能开发并行。资料只用于经确认的项目范围，不进入 GitHub 仓库。

## 2. 交付原则

- 先小批量验证格式和流程，再按病种分批扩展，不必一次交齐全部资料。
- 病例必须先脱敏；教材和指南必须确认授权或项目内合理使用边界。
- 每批资料由一名医学联系人确认完整性，并指定可答疑和审核的专业人员。
- 资料原件、转换件和运行产物放在本机私有数据目录，禁止放入 `wiki/` 之外的受版本管理目录。
- 文件名、表格和正文不得保留真实姓名、联系方式、证件号、住院号等直接标识。

## 3. S2 最小输入包

首次真实生成验证需要同一个试点病种的以下内容：

1. 至少 1 份当前认可的指南或教材节选，附标题、版本或年份、适用范围和授权说明。
2. 至少 3 例结构完整、已脱敏病例，覆盖典型表现和至少一种需要讨论的差异点。
3. 目标学习者说明，例如年级、专业背景和已有知识。
4. 3–5 条可观察的教学目标，使用“识别、解释、选择、比较”等动作表达。
5. 每例期望诊断、关键线索、核心问题、参考答案要点和禁用或高风险表述。
6. 一名负责答疑与最终医学审核的专业人员，以及可预约的审核时间窗口。

缺少指南/教材依据、合格脱敏病例、教学目标或医学审核人会阻塞 S2 的真实病例生成闭环。格式不统一通常不阻塞，我们会先用转换脚本适配并反馈标准模板。

## 4. 后续病种输入包

S3 以后按病种逐批提供，每批建议包含 5–10 例脱敏病例、相应教材或指南、教学目标、难度标注建议和审核人。只有计划进入当前迭代的病种才有最晚交付时间，其余病种准备可持续并行。

最终语料目标与发布包目标不同：语料可用于筛选和生成，只有通过质控与医学审核的内容才成为发布包。

## 5. 病例建议结构

每例建议使用统一表格或 Markdown，包含：

- 项目内病例编号，不使用医院原编号。
- 病种、年龄段、性别（如有教学必要）。
- 主诉、现病史、既往史、用药史和必要的个人史。
- 体格检查、实验室与影像摘要，注明单位和参考范围。
- 诊断、鉴别诊断、诊疗关键点及教学难点。
- 已做的脱敏处理和资料提供者确认日期。

不得为了“完整”补造原病例中不存在的医学事实。不确定或缺失内容应明确标注。

## 6. 脱敏检查

交付前至少检查并移除或泛化：姓名、证件号码、详细地址、联系电话、电子邮箱、住院号/门诊号、精确就诊日期、罕见职业或事件组合、影像角标和附件元数据中的身份信息。

脱敏并非只删除姓名。若多个间接信息组合后仍容易识别个人，应进一步分箱、偏移日期或删除非教学必要信息，并由医学联系人复核。项目负责人保留脱敏确认记录，但身份映射不得交给开发目录或商业 API。

## 7. 资料与授权元数据

每项教材或指南同时提供：资料编号、标题、发布机构、版本/年份、来源链接或受控存放位置、允许用途、摘录限制和确认人。无法确认授权的全文不导入系统，可只记录公开链接和允许使用的必要摘录。

病例资料记录来源批次、是否为合成、脱敏完成状态、允许的处理边界和保留期限。授权范围变化应立即通知项目负责人。

## 8. 交付目录与安全

推荐在仓库外或仓库已忽略的私有数据根目录中使用：

- `private-data/knowledge/`：教材与指南。
- `private-data/cases/<disease-code>/`：脱敏病例。
- `private-data/intake/`：待检查的新资料。
- `private-data/review/`：医学审核交换文件。

具体本机路径通过 `.env` 配置，不写进代码和提交历史。发送商业模型前，系统和操作者都要确认该批资料允许调用外部 API。

## 9. 并行计划

- S0–S1 开发期间：医学团队可并行确认首个试点病种、资料清单、脱敏负责人和审核人，不阻塞合成数据骨架。
- S2 开始前：完成最小输入包，否则真实模型与检索验证阻塞；工程侧仍可继续完善合成流程和界面。
- S2 进行中：医学团队审核前 3 个结果并反馈模板，其他病种可继续准备。
- S3–S6：按下一个迭代病种提前一阶段交付；无需等待全部病种。
- S7 以后：提供正式评价专家、冻结评价集并按统计方案执行。

## 10. 每批交付验收

项目负责人按以下清单确认：数量达到当前批次要求；文件可读；字段基本完整；脱敏已签认；授权状态明确；病种和教学目标一致；医学联系人与审核时间可用。未通过项登记到输入台账，并标明“阻塞、限期补充或可后补”。
